# Pre-1.22 version compatibility: feasibility measurement and shim design

Date: 2026-07-12
Status: feasibility measured (this pass); implementation not started
Tracks: issue #15 "extend version compatibility below the 1.22.0 floor"
Game versions probed: 1.21.7 and 1.20.12 (compiled against, decompiled, and run live),
1.22.3 as the reference (decompiled); 1.19.8 and 1.18.15 sized from the recorded
compatibility sweep only (no local install, not re-measured)
Prerequisites: [Atlas design](2026-07-02-atlas-design.md),
[world snapshot/rollback](2026-07-06-world-snapshot-rollback.md),
[rollback stage 3](2026-07-11-rollback-stage3-mod-cooperation.md)

## Motivation

The README compatibility table records 1.22.0 as the supported floor, with every older
line failing at build time. Issue #15 was closed for lack of demand and has been reopened:
broader version compatibility is now wanted first-party, for differential testing (the
same scenario suite against several engine versions) and for ecosystem mods that still
target the 1.20.x/1.21.x server lines. The archived EngineCompat sketch in the issue's
closing comment is the starting idea; the harness surface has since grown rollback
(stages 1-3), kicked-player cleanup, the assets-build settle probe, and the boot
preflights, all of which touch engine internals. This pass measures, per version, what
actually breaks, what a shim must cover, and what should be gated off, so the floor
decision is grounded in evidence rather than in the 2026-07-03 sweep's one-line verdicts.

## Method

All measurements were made on this branch with a temporary reflective probe standing in
for the 1.22 exit-lifecycle API (the archived sketch's shim, reduced to its two members),
so that second-wave compile errors hidden behind the first ones could surface:

- `dotnet build src/Atlas/Atlas.csproj` against each version's own dlls
  (`VINTAGE_STORY=~/dev/.vs-compat/<version>`), pristine and with the probe.
- Decompilation of `VintagestoryLib.dll` 1.21.7 (full), 1.20.12 and 1.22.3 (targeted)
  with ILSpy, comparing every engine member Atlas touches, reflectively or directly.
- Live runs of the real E2E suite subsets (smoke boot, world rollback, player rollback,
  player join) against 1.21.7 and 1.20.12, with the probe in place.

The probe itself was deleted after measurement; this document is the only artifact.

## Measured API differences

### 1.21.7 and 1.20.12: the exit lifecycle is the entire compile-level gap

A pristine build against either version fails with exactly four distinct errors (eight
raw error lines, msbuild reporting each twice), all in
`src/Atlas/Internal/Hosting/ServerHost.cs`, all on the 1.22 exit-lifecycle API:

| Error | Site | Missing symbol |
|---|---|---|
| CS0103 x2 | `server.Stop(reason, EnumExitMode.SoftExit)` (normal and crash teardown) | `EnumExitMode` |
| CS0117 | `new ServerMain(...) { exitState = ... }` | `ServerMain.exitState` |
| CS0246 | `new GameExitState()` | `GameExitState` |

With those four references routed through a reflective shim, the build is clean on both
versions: zero errors, zero warnings, with `TreatWarningsAsErrors` on, for the full
project graph (Atlas, Atlas.Bridge, Atlas.XUnit, Atlas.Cli, and every test project).
Everything else Atlas compiles against directly (the rollback stack over `GameDatabase`,
`ChunkServerThread`, `MagicNum`, `SaveGame`, `GameCalendar`; the player stack over
`ConnectedClient`, the dummy socket classes, `Packet_Client`; `ServerProgramArgs`,
`StartServerArgs`, `ServerLogger`, `Lang.PreLoad`; `ServerWorldPlayerData`'s full scalar
surface including `DidSelectSkin` and `SpawnPosition`; `SaveGame.MiniDimensionsCreated`;
`LoadChunkColumnForDimension`) exists with a compatible signature on 1.21.7 and 1.20.12.

The pre-1.22 equivalent, from the decompile (identical on 1.21.7 and 1.20.12):

- `public GameExit exit;` on `ServerMain`, where 1.22 has
  `public GameExitState exitState;`. `GameExit` is a public, parameterless-constructible
  class (global namespace) holding a single bool with `SetExit`/`GetExit`.
- `public void Stop(string reason, string finalLogMessage = null, EnumLogType
  finalLogType = EnumLogType.Notification)`, where 1.22 has
  `Stop(string reason, EnumExitMode exitMode)`.
- The engine never assigns `exit` itself (only the dedicated entry point does). On an
  embedded boot it stays null and is dereferenced in three places:
  `ProcessNetMessage` checks `exit.exit` on every inbound packet, `Stop` calls
  `exit.SetExit(true)`, and the `clientPacketsParser` background thread (spawned by
  `PreLaunch()` unless `ReducedServerThreads`) reads `server.exit.exit` every 10 ms from
  boot. 1.22 is structurally identical with `exitState.Mode`/`exitState.Exit`, which is
  precisely why Atlas installs `exitState = new GameExitState()` before `PreLaunch()`
  today. The same install, into whichever field the loaded engine has, is the whole fix.

Two findings refine the archived sketch:

- `ReducedThreads` is not needed. The sketch set `ServerProgramArgs.ReducedThreads = true`
  on 1.21.x, reporting a boot NRE from the parser thread. The thread's loop only
  dereferences `exit`, and with the holder installed before `PreLaunch()` (the same
  ordering Atlas already uses on 1.22) the live smoke runs below passed on 1.21.7 and
  1.20.12 with the thread running. The sketch's ServerHost predated the 0.5.0/0.6.0
  rewrites; its NRE is best explained by a holder installed too late. `ReducedThreads`
  exists on 1.20.12 through 1.22 as an option if determinism ever wants it, but it would
  change 1.22 behavior too and is not part of this design.
- The `GameVersion` const trap is real and measurable. `GameVersion.NetworkVersion` is a
  `const`: "1.22.6" on 1.22.3, "1.21.10" on 1.21.7, "1.20.8" on 1.20.12. Consts are baked
  into the referencing assembly's IL at compile time, and the server hard-rejects a
  mismatch in `HandlePlayerIdentification` (1.21.7:
  `if ("1.21.10" != identification.NetworkVersion) DisconnectPlayer(...)`). A rebuilt
  Atlas (the sweep model) bakes the right value; a prebuilt Atlas.dll (the NuGet model)
  bakes the version it was compiled against and every `JoinPlayer` on an older engine
  is rejected. The shim reads both constants from the loaded assembly's metadata
  (`GetRawConstantValue`) instead.

|---|---|---|
| SmokeTests (boot, tick, assert, stop; twice in-process) | 2/2 pass | 2/2 pass |

| AdapterJoinPlayerTests (`EntitiesIn` sees the joined entity) | 0/1 fail | 0/1 fail |

That is: boot, the exit lifecycle, host recycling, the assets-build settle, the full

save-idle reflection), the dummy-socket join, inventories, teleports, per-player
moddata, and player-aware rollback all work on 1.21.7 today, behind a two-member
reflective shim. 1.20.12 matches everywhere it was exercised.

### The one measured runtime difference: `Entity.Pos` vs `ServerPos`, root-caused

`JoinPlayer` succeeds on both old versions (entity spawned, inventories wired), but
`World.EntitiesIn(player.Position.Area(16))` comes back empty. A runtime probe
pinpointed it. On 1.21.7 the joined player reports `state=Active`, is present in
`LoadedEntities`, and is registered in the correct spawn chunk (`InChunkIndex3d`
matches the spawn column), but `Entity.Pos` reads (0, 0, 0) while `ServerPos` holds the
real spawn position; on 1.22.3 the same probe reads the real position from `Pos` and
finds the entity. The cause is an API unification in 1.22:

- 1.22.3: `public EntityPos Pos { get; private set; }` with `ServerPos => Pos` and
  `SidedPos => Pos` (one instance, three names).
- 1.21.7 and older: `Pos` and `ServerPos` are two separate `EntityPos` instances; the
  server's join path only sets `ServerPos` before `SpawnEntity`, and nothing updates a
  headless player's `Pos` until something like a teleport writes it.

So the query was never broken: `GetEntitiesInsideCuboid` is line-identical across
versions and internally uses `SidedPos`. Atlas fed it the wrong center, because
`TestPlayer.Position` reads `Entity.Pos`. The fix is mechanical and version-neutral:
read and write server-side entity positions through `SidedPos` (returns `ServerPos` on
the server on 1.21.x, `Pos` on 1.22, verified in both decompiles) or `ServerPos`.
Affected Atlas sites, all of the same family: `TestPlayer.Position`,
`TestPlayer.TeleportTo`'s dimension check, `WorldSession.SpawnEntity` (it sets
`entity.Pos`, but pre-1.22 `SpawnEntity_internal` reads `entity.ServerPos` for chunk
registration, so scenario-spawned entities would land in the chunk at the origin), and
`PlayerRollbackState`'s position capture/restore (its "Pos IS ServerPos" comment is a
1.22-only fact). The player-rollback suite passed on 1.21.7 despite this because
capture and assertion looked through the same lens; the fix makes the captured
positions real instead of self-consistent.

## Shim architecture: the EngineCompat layer

One internal static class, `EngineCompat`, owning every touchpoint whose shape varies
across supported versions. Three tiers, in strict preference order:

1. **Reflective, boot-validated** (the default, and everything 1.21/1.20 needs):
   - `InstallExitState(ServerMain)`: `GetField("exitState") ?? GetField("exit")`,
     instantiate the field's type, assign. Called where the object initializer sits
     today, before `PreLaunch()`.
   - `Stop(ServerMain, string reason)`: adapt to whichever `Stop` signature the loaded
     engine has (two parameters with an enum second: 1.22, invoked with `SoftExit`;
     otherwise the 1.21-and-older three-parameter form).
   - `NetworkVersion` / `ShortGameVersion`: read from the loaded assembly's metadata,
     used by `DummyClientConnector` instead of the baked consts.
   Handles are resolved once per process in static initializers and every failure throws
   with the game version in the message (the same fail-fast discipline
   `WorldSnapshot.Create` and the assets-probe warning already follow). A version whose
   layout drifts fails at boot with a named symbol, never mid-scenario.
   Source rule that makes the single binary work: outside `EngineCompat`, Atlas source
   never mentions a type or member that does not exist on every supported version, so no
   Atlas method can fail to JIT on an older engine. The zero-error builds from identical
   source on 1.20.12 through 1.22.3 are the proof this rule currently holds; the sweep
   keeps it true.

2. **Compile-time conditional**: not used. It would only become necessary at 1.19.8,
   where the recorded sweep says `PreLaunch`/`ServerProgramArgs` themselves change shape,
   which reflection over an otherwise-identical boot cannot paper over. Crossing that
   line forks the shipped artifact per version family and multiplies the pack matrix;
   this design deliberately stops above it.

3. **Gated-off features**: the existing rollback degrade machinery
   (`RollbackUnsupportedException`, `RollbackDegradeReason.EngineDrift`, fail-closed
   recycle fallback) already is the gate for anything that goes missing on an engine:
   nothing new is needed for 1.21/1.20, where every prerequisite verified present. Gating
   is the planned answer for any FUTURE version that drops a prerequisite, not for the
   old ones measured here.

## Per-feature support matrix

"Supported": works through the common code path, verified. "Shimmed": works through
`EngineCompat`, verified live this pass. "Expected": decompile-verified parity, not
exercised live this pass. "Gated": build fails today; needs work beyond this design.

| Feature | 1.22.x | 1.21.7 | 1.20.12 | 1.19.8 | 1.18.15 |
|---|---|---|---|---|---|
| Boot + scenarios (tick pump, `Api`, commands) | supported | shimmed | shimmed | gated (boot signature fork) | gated (same + `Func` alias) |
| Exit lifecycle / clean teardown | supported | shimmed | shimmed | gated | gated |
| Assets-build settle probe | supported | expected (shape verified) | expected | unknown | unknown |
| Preflights (pdb, api-copy) | supported | supported (version-agnostic) | supported | supported | supported |
| Test players: join, inventories, `GiveItem`, `TeleportTo`, moddata | supported | shimmed | expected (join path identical; smoke not run) | gated | gated |
| `EntitiesIn` / `Position` of joined players | supported | broken, root-caused (`Pos` vs `ServerPos`; mechanical `SidedPos` fix) | broken, same cause | same cause expected | same cause expected |
| Kicked-player cleanup | supported | expected (all four structural facts verified) | expected | unknown | unknown |
| Fixtures, schematics, data seeding | supported | expected (API-only surface) | expected | unknown | unknown |
| Rollback stage 1 (world) | supported | shimmed, verified live | shimmed, verified live | gated | gated |
| Rollback stage 2 (players) | supported | shimmed, verified live | expected | gated | gated |
| Rollback stage 3 (mini-dimensions) | supported | expected (every prerequisite verified: dimension-keyed rows, `TryUnloadChunk`, `LoadChunkColumnForDimension`, partial-column discard) | expected | unknown (mini-dimension era boundary) | unlikely |

## CI story

The sweep (`compat.yml`) already implements the right model: download the version's
server, build the whole solution against ITS dlls, run the full E2E suites, emit a
compatible/partial/incompatible verdict. Atlas compiles against the install's dlls by
design, so "supported version" means "green in that lane", and the matrix is data
(`prepare.compute` emits the version list).

Growth plan, in cost order:

1. Keep the weekly sweep as the oracle for the full historical matrix (it already runs
   1.18.15 through 1.22.3; each job is bounded at 20 minutes, servers are cached).
2. When the floor moves, promote the new floor version into the per-push E2E matrix next
   to latest 1.22.x: two full E2E lanes per push instead of one. Per-push cost roughly
   doubles; that is the honest price of a supported floor, and it is what keeps the
   README table truthful ("tested on every push" vs "swept weekly").
3. The prebuilt-binary story needs its own lane eventually: pack Atlas against 1.22.x,
   then run the E2E suite on 1.21.7 loading the PACKED dll (not a rebuild). The
   reflective shim plus the const-metadata reads are exactly what make this lane
   passable; nothing else in this design is validated for it yet, so it ships in the
   last stage, not the first.

## Floor recommendation

**Adopt 1.21.x as the new floor first** (confirming the prior). Measured cost: one
reflective shim (two members plus two metadata reads, roughly the 70-line archived
sketch), the mechanical `SidedPos` position fix at four named sites, one added per-push
CI lane. Every heavyweight subsystem (boot, players, both shipped rollback stages) is
already verified live behind the probe. 1.20.12 measures identically at the compile
level and passed everything it was exercised on, so **1.20.x is a cheap fast-follow**
once the `SidedPos` fix proves out and one full-suite sweep run is green there; it
should not gate the 1.21.x landing.

**1.19.8 and 1.18.15 stay unsupported** (refuting any "go all the way down" reading):
the boot path itself changes shape (compile-time forks, a per-version artifact matrix),
the mini-dimension and rollback surface there is unmeasured, 1.18.0-1.18.7 ship a
different server archive entirely, and no consumer has asked. Reopen per version on
concrete demand, with this document's method as the template.

## Staged plan

- **Stage 0, EngineCompat**: land the shim (exit install, `Stop` adaptation, version
  metadata reads) with boot validation; route `ServerHost` and `DummyClientConnector`
  through it. Zero behavior change on 1.22 (same install ordering, same `SoftExit`),
  proven by the existing per-push E2E lane.
- **Stage 1, 1.21.x floor**: apply the `SidedPos` fix at the four sites named above and
  re-run the join suite on 1.21.7 AND 1.22.x (the fix must be a no-op there); add
  1.21.7 to the per-push matrix; move the README floor to 1.21.x with the support
  matrix above, including the honest per-feature notes.
- **Stage 2, 1.20.x floor**: run the full sweep lane on 1.20.12 including
  `Sample.Scenarios` and the player/fixture/hook suites; if green, add to the README
  table (swept weekly, not per push, unless demand says otherwise).
- **Stage 3, binary lane (optional, on demand)**: the packed-dll-on-old-engine CI lane;
  only this stage claims NuGet-binary compatibility below 1.22.
- **Not planned**: 1.19.x and below.

## Maintenance costs, honestly

- **The shim is invisible to the compiler.** Reflection trades compile errors for boot
  errors. The boot-validated fail-fast keeps the failure loud and named, but only the
  sweep actually proves each engine still has the members: the weekly cadence is the
  real safety net, and a red weekly run must be treated as a release blocker for the
  affected line.
- **Every new engine touchpoint inherits a discipline.** Whoever adds one must ask "does
  this exist on the floor version?" and route it through `EngineCompat` (or the rollback
  degrade path) when in doubt. The source rule (no 1.22-only symbols outside the shim)
  is enforceable only by building against the floor, so the floor build should join the
  per-push matrix, not just E2E.
- **CI wall time scales linearly with supported lines.** One per-push lane per supported
  floor line, each booting real servers. The measured suites ran in seconds-to-minutes
  here, but the full matrix job budget is 20 minutes per version and will be paid on
  every push.
- **Old lines are frozen upstream**, which caps the drift risk: 1.21.7 and 1.20.12 will
  never change again, so their verification is one-time plus re-runs, not a treadmill.
  The treadmill risk stays where it always was, on NEW game versions.
- **The support matrix becomes a public promise.** Per-feature footnotes (for example
  the `EntitiesIn` note until fixed, the binary-lane caveat until stage 3) must live in
  the README table, or "compatible" quietly overclaims. Keeping that table synchronized
  with the sweep verdicts is a small recurring documentation cost.
